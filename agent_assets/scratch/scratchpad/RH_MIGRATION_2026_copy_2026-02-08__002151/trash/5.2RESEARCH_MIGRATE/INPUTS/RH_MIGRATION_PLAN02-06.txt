SECTION A: THE MENTAL MODEL
The Problem (one paragraph, no fluff)

You have ~60,000 personal files scattered across 5+ competing root structures on C:. Your real work (detections, playbooks, automation, proof builds) exists as scattered artifacts instead of coherent work units. A Wazuh rule lives in one folder, the screenshot proving it fired lives in another, the playbook in a third. You can’t find, prove, or build on your own work. Every cleanup attempt generates more metadata without finishing the migration, creating a loop. This plan breaks the loop by making all decisions up front, using automated discovery to cluster files into work units, requiring human review before any move, and installing guardrails so chaos can’t regenerate.

Core Principle: Work Units, Not File Types

Wrong approach: “Put all .yml here, all .md there.”
Right approach: “Everything related to my Wazuh baseline work goes into one proof pack.”

A work unit = a cluster of files that belong together because they’re part of the same thing you did. Detection rule + playbook + screenshots + deployment notes = one proof pack.

Operating Model for Every Phase

Every destructive operation follows this cycle:

PLAN → DRY RUN → HUMAN REVIEW → EXECUTE → VERIFY


No exceptions. Models generate scripts and reports. You approve and run them. Nothing moves without a log.
